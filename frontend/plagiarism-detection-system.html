Project Modules

What the system does

Submission management

Stores students, courses, assignments, and one controlled submission per student per assignment to avoid duplication.

Token-based comparison

Breaks each submission into tokens and uses overlap counts to compute similarity percentages between submission pairs.

Faculty review support

Generates plagiarism reports, marks high-risk records, and exposes clean views for quick inspection during academic review.

Execution Flow

From submission to report

1

Store data

Faculty, courses, students, assignments, and submissions are inserted into normalized relational tables.

2

Tokenize text

The tokenization procedure splits submission text into words and updates token frequencies plus word count.

3

Calculate similarity

Functions and procedures compare common tokens and calculate similarity using the smaller token count as denominator.

4

Flag suspicious pairs

Triggers mark risky submissions and views surface plagiarism reports for evaluator-facing interpretation.

Database Design

Core schema of the project

The structure separates academic data, textual analysis, and reporting logic so the evaluator can clearly see both normalization and system behavior.

Academic layer

  • Faculty manages courses.
  • Students are linked to courses.
  • Assignments belong to courses.

Analysis layer

  • Each submission stores text, timestamp, word count, and flag state.
  • Tokens preserve extracted words and frequencies for comparisons.

Reporting layer

  • Plagiarism reports store compared submission pairs and similarity percentages.
  • Views summarize high-risk pairs and submission statistics.
